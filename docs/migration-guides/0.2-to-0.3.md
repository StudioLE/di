# Migration Guide: 0.2 to 0.3

## Migrating from `Service` to `FromServices`

This is a breaking redesign. The `Service` trait, derive macro, and async-first construction have been replaced with a sync-first `FromServices` trait, explicit scope control, and trait object resolution support.

## Quick Reference

| Before                             | After                                                        |
|------------------------------------|--------------------------------------------------------------|
| `Service` trait                    | `FromServices` trait                                         |
| `#[derive(Service)]`               | `#[derive(FromServicesAsync)]`                               |
| `ServiceError`                     | `ResolveError`                                               |
| `ServiceProvider::new()` (builder) | `ServiceBuilder::new()...build()`                            |
| `with_instance(value)`             | `with_instance(value)`                                       |
| `with_factory(f)`                  | `with_type::<T>()`                                           |
| `get_service::<T>().await?`        | `get::<T>()?`                                                |
| N/A                                | `with_trait::<dyn Foo, Impl>()` (nightly, `traits` feature)  |
| N/A                                | `get_trait::<dyn Foo>()?`                                    |

## Features

| Feature  | Requires | Adds                                                                   |
|----------|----------|------------------------------------------------------------------------|
| (none)   | stable   | `FromServices`, `with_type`, `with_instance`, `get`                    |
| `async`  | stable   | `FromServicesAsync`, `with_type_async`, `get_async`, `get_trait_async` |
| `traits` | nightly  | `with_trait` (clean trait registration via `Unsize` coercion)          |

## Step-by-step

### 1. Update the derive macro

`#[derive(Service)]` has been renamed to `#[derive(FromServices)]`. Fields marked `#[di(default)]` are initialized via `Default::default()` instead of being resolved from the container. For async construction, use `#[derive(FromServicesAsync)]`.

Before:

```rust
#[derive(Service)]
pub struct MyHandler {
    http: Arc<HttpClient>,
    db: Arc<Database>,
}
```

After:

```rust
#[derive(FromServices)]
pub struct MyHandler {
    http: Arc<HttpClient>,
    db: Arc<Database>,
}
```

Or implement `FromServices` manually for more control:

```rust
impl FromServices for MyHandler {
    type Error = ResolveError;

    fn from_services(services: &ServiceProvider) -> Result<Self, Report<ResolveError>> {
        let http = services.get::<HttpClient>()?;
        let db = services.get::<Database>()?;
        Ok(Self { http, db })
    }
}
```

### 2. Replace `Service` trait with `FromServices`

The `Service` trait was async. `FromServices` is sync, and both traits have an associated `Error` type.

Before:

```rust
impl Service for Database {
    type Error = ServiceError;

    async fn from_services(
        services: &ServiceProvider,
    ) -> Result<Self, Report<Self::Error>> {
        let config = services.get_service::<Config>().await?;
        Ok(Self { config })
    }
}
```

After:

```rust
impl FromServices for Database {
    type Error = ResolveError;

    fn from_services(services: &ServiceProvider) -> Result<Self, Report<ResolveError>> {
        let config = services.get::<Config>()?;
        Ok(Self { config })
    }
}
```

If your service genuinely needs async construction, enable the `async` feature and implement `FromServicesAsync` instead. When construction involves fallible async operations beyond DI resolution, use a domain error type:

```rust
impl FromServicesAsync for Database {
    type Error = DatabaseError; // your domain error type

    async fn from_services_async(services: &ServiceProvider) -> Result<Self, Report<DatabaseError>> {
        let config = services.get::<Config>().change_context(DatabaseError::Resolve)?;
        let connection = connect_to_db(&config).await?;
        Ok(Self { config, connection })
    }
}
```

### 3. Update ServiceProvider construction

Construction now uses `ServiceBuilder` with a consuming builder pattern, finalized by `.build()`.

Before:

```rust
let services = ServiceProvider::new()
    .with_instance(config)
    .with_factory(|sp| Database::new(sp));
```

After:

```rust
let services = ServiceBuilder::new()
    .with_instance(config)
    .with_type::<Database>()
    .build();
```

### 4. Singleton vs transient

Services are singletons by default. Use the `_transient` variants for a fresh instance on every resolution:

- `with_type::<T>()` - singleton (default, same as old behaviour)
- `with_type_transient::<T>()` - fresh instance each time

### 5. Replace `get_service` with `get`

Before:

```rust
let db: Arc<Database> = services.get_service::<Database>().await?;
```

After:

```rust
let db: Arc<Database> = services.get::<Database>()?;
```

No `.await` - resolution is sync. For async services, use `get_async`:

```rust
let db: Arc<Database> = services.get_async::<Database>().await?;
```

### 6. Update error types

Before:

```rust
ServiceError::NoService
ServiceError::Create
```

After:

```rust
ResolveError::NotFound
ResolveError::Factory
```

## New: Trait Object Resolution

*Requires nightly + `traits` feature*

Enable the feature in `Cargo.toml`:

```toml
[dependencies]
studiole-di = { version = "...", features = ["traits"] }
```

Add `rust-toolchain.toml` to your project root:

```toml
[toolchain]
channel = "nightly"
```

Register a concrete type and resolve it as a trait object:

```rust
let services = ServiceBuilder::new()
    .with_trait::<dyn Get, MemoryCache>()
    .with_trait::<dyn Set, MemoryCache>()
    .build();

let cache: Arc<dyn Get> = services.get_trait::<dyn Get>()?;
```

Both trait registrations share the same concrete singleton.

### Swapping implementations in tests

```rust
let services = ServiceBuilder::new()
    .with_trait::<dyn Get, MockCache>()
    .build();
```

## New: Async Resolution

Enable the `async` feature (works on stable):

```toml
[dependencies]
studiole-di = { version = "...", features = ["async"] }
```

Register with `with_type_async` and resolve with `get_async`:

```rust
impl FromServicesAsync for Database {
    type Error = ResolveError;

    async fn from_services_async(services: &ServiceProvider) -> Result<Self, Report<ResolveError>> {
        let config = services.get::<Config>()?;
        Ok(Self { config })
    }
}

let services = ServiceBuilder::new()
    .with_type_async::<Database>()
    .build();

let db = services.get_async::<Database>().await?;
```

`get_async` resolves both sync and async registrations. Calling sync `get` on an async registration returns `ResolveError::Async`.
